# CRO Test Report

Replaces the Data Studio per-test dashboards. One self-contained HTML file per test:
A/B Tests tab (default, with significance), Overview, Acquisition, Click Tracking and
Form Tracking tabs (the last two auto-hide until Phase-2 queries exist). All filters are
real — every scorecard, table, and the z-test recompute from an embedded fact table.

**Pipeline per test:** Asana task → dates + IDs → GA4 pulls → `build_report.py` → HTML.

## Bundled files

| File | Role |
|---|---|
| `assets/report_template.html` | The dashboard. Layout + filter engine. Brands itself per school (RAS green / AMU black+gold / APU black+cyan) from DATA.meta.school — no per-run styling needed. NEVER edit per-run; the data block between `/*__DATA_START__*/` and `/*__DATA_END__*/` is regenerated by the build script. |
| `scripts/build_report.py` | Deterministic transform: config + GA4 pull JSON → final HTML. Has a verification gate (exits non-zero on zero conversions, missing audiences, sum mismatches). Use it for ALL builds — don't hand-splice data. |
| `references/ga4_queries.md` | The three required GA4 queries (exact dims/filters), event-discovery procedure, MCP failure playbook, and the direct-API PowerShell fallback. Read before pulling. |
| `references/event_map.json` | Per-school property / hostname / RFI / App event names. All three schools VERIFIED 2026-07-15 (RAS, AMU, APU) — see per-school notes for the AMU/APU abConv hostname-filter and audience-suffix caveats. |
| `references/example_config_10464.json` | Known-good config for the reference test. |

## How a request arrives

**Pattern A — Single test by name or number.** "get a test report for 10464",
"report for the ECE redesign test":
1. `Asana:search_objects` / `search_tasks` on the **2026 CRO APEI Board**
   (GID `1214706069876945`) using the test name or number.
2. ⚠ **RH#s are NOT unique** — multiple tasks can share "(10464)". Disambiguate by
   school + page path / Control URL. If still ambiguous, ask; don't guess.
3. Read custom fields: **Live Date**, **End Date**, **Optimizely Project ID**,
   **Optimizely Experiment ID**, **Control URL** (or the URL in the task name).

**Pattern B — Board section.** "run test reports for the Live column":
1. `get_project` with sections on the board. Known: **Live = `1214709647758784`**.
2. `get_tasks` on the section with
   `opt_fields=name,start_on,due_on,custom_fields,custom_fields.name,custom_fields.display_value`.
3. Loop Pattern A steps 2–3 per task; produce one report per test. Summarize all
   tests in chat at the end (one line each: window, sessions, RFI lift, verdict).

**Pattern C — Multiple named tests.** Same as A per test.

## Date rule (authoritative)

- Window = Asana **Live Date → End Date**.
- **End Date empty → Live Date → yesterday** (today − 1; never include the partial
  current day).
- Live Date empty → ask the user; do not infer from Optimizely or GA.
- Never use a Data Studio report's date range — DS ranges have included weeks of
  post-test data before.

## GA4 pull

Read `references/ga4_queries.md`, then run the three queries (abTraffic, abConv,
abDaily) with:
- property + hostname + event names from `references/event_map.json` for the task's
  school (UNVERIFIED school → run event discovery first and confirm with the user,
  then update event_map.json);
- audience match token = page-path fragment + `(<RH#>)`, e.g. `"overview (10464)"` —
  never the RH# alone;
- the Asana date window.

Access hierarchy: **GA MCP → direct Data API (PowerShell fallback in ga4_queries.md)
→ Claude in Chrome.** Windsor.ai is retired — never reference or attempt it.
Zero audiences returned → Optimizely audience-name fallback (see ga4_queries.md).
Personalization audiences of the same test are included as extra variation rows.

## Build

1. Write `config.json` (copy example_config, set title/testId/school/dates/events).
   Title convention: `<Test Name> A/B Test Report` from the Asana task name's
   test segment.
2. Save the pull as one JSON with keys `abTraffic`, `abConv`, `abDaily`
   (MCP responses or the fallback script's output — build_report reads BOM'd UTF-8).
3. `python3 scripts/build_report.py --config config.json --pull pull.json --out
   <school>_<testId>_<slug>_report.html`
4. The script prints per-audience aggregates. **Verify before delivering:** sessions
   per arm are plausible vs. Optimizely, RFI/App non-zero, control listed first.
   If the verification gate exits non-zero, fix the cause — never ship anyway.

## Deliver

- Output file(s) to the user (present/send). Filename: `ras_10464_ece_redesign_report.html` style.
- Chat recap per test: window (+ source), sessions per arm, RFI CVR lift + one-sided
  confidence vs. the 90% ship rule, App Complete direction, and any notable segment
  (e.g., mobile-driven lift) found by toggling the device filter mentally from the facts.
- The user maintains a GitHub Pages report hub — after delivering the file, prepare the
  publish payload (HTML + reports.json entry + commit message) per
  `references/report_hub.md`.

## Guardrails

- The template's layout/engine code is battle-tested — regenerate ONLY the marker
  block via build_report.py.
- Users metric is non-additive across segments; the dashboard already labels it
  approximate under filters. Don't "fix" it by re-pulling per-combination.
- Report scope = the test's audiences, not the whole site (~sitewide is 25× larger;
  a giant sessions number means the audience filter was dropped).
- If numbers differ from an old Data Studio report, check the date window first —
  that's the usual reason, and Asana wins.
